Overview

The Build Dashboard is a Flask-based build tool that compiles a Python Flask application into a standalone Windows installer. It integrates Nuitka or PyInstaller for bundling, optional PyArmor source obfuscation for code protection, and Inno Setup for creating a professional .exe installer — all driven from a single browser-based interface.

What it automates
  • Git branch management (dev → main)
  • Dependency installation & verification
  • Source obfuscation or native compilation
  • PyInstaller / Nuitka bundling
  • Inno Setup installer creation
  • Scheduled source backups via robocopy
What you do
  • Fill in App Info fields once per project
  • Pick a protection strategy
  • Click Full Build and watch the pipeline
  • Distribute the .exe installer from build_output/

Launching the Build Dashboard

The Build Dashboard runs on port 5001 and is launched via the PowerShell script start_builder.ps1 located in the project root. Double-click it in Explorer, or run it from a terminal:

Terminal
PS C:\...\job_tracker> .\start_builder.ps1
# Opens browser at http://127.0.0.1:5001

The script starts start_builder.py, which in turn launches build/builder_app.py. The Flask server runs in the background and your default browser opens automatically. Close the terminal window (or press Ctrl+C) to stop the server.

Keep it open: The Build Dashboard does not need to be relaunched between builds. You can start it once in the morning and leave it running all day.

Project Profiles

A Profile is a named snapshot of every configuration field in the Build Dashboard — App Info, protection settings, Git paths, backup directories, and daemon settings. Profiles let you switch between multiple projects instantly without re-entering settings each time.

Profile selector bar (top of dashboard)
Profile:

Actions

  • Save — Writes the current configuration to build_settings.json under the profile name you type in the prompt. If a profile with that name already exists it is overwritten.
  • Load — Restores all fields from the selected profile. Any unsaved changes to the current configuration are discarded.
  • Delete — Permanently removes the selected profile from build_settings.json. A confirmation dialog is shown first.
build_settings.json is machine-specific and is excluded from Git (.gitignore). It lives inside the build/ folder and is never committed to the repository.
Create one profile per project you maintain with the Build Dashboard. Name them descriptively, e.g. job_tracker-v2.1 or myapp-staging.

Fields reference

  • App Name — The human-readable product name shown in the installer wizard title bar and Windows "Add/Remove Programs".
  • Version — Semantic version string (MAJOR.MINOR.PATCH). Inno Setup embeds this in the installer metadata and uses it for upgrade detection.
  • Description — One-line description included in the installer and optionally in the compiled executable's version info block.
  • Publisher — Developer or company name shown in the installer and in Windows program listings.
  • App URL — Project homepage. Shown on the installer "Welcome" page and recorded in the registry entry.
  • Support URL — Link to your issue tracker or support page. Shown in "Add/Remove Programs" details.
  • Exe Name — The base filename of the compiled executable (without .exe). Must be a valid Windows filename with no spaces.
  • Output Dir — Root folder where all build artefacts are written. Defaults to D:\Compile Playground\<RepoName> (derived from the Working Tree Path). Three sub-locations are used automatically:
    • <OutputDir>\ — The finished installer .exe goes here (e.g. JobTracker_Setup_2.1.0.exe).
    • <OutputDir>\Bundle Only\<ExeName>\ — PyInstaller / Nuitka bundle folder. The compiled executable and all its dependencies live here.
    • <OutputDir>\Log\ — Timestamped build logs and test-pipeline logs are written here.
    All three directories are created automatically if they do not already exist. Click the button to open the folder in Windows Explorer.
  • Install Dir — Default installation path suggested to the end user, e.g. {pf}\Job Tracker. Inno Setup macros like {pf} (Program Files) are supported.
  • Require Admin — Toggle. When enabled, the installer requests UAC elevation. Required if you use {pf} or write to HKLM registry keys.
  • Desktop Shortcut — Adds a shortcut to the user's Desktop after installation.
  • Start Menu Shortcut — Creates a Start Menu entry under Publisher\App Name.
  • Startup Shortcut — Adds a shortcut to the Windows Startup folder so the app launches on login.
  • Icon File — Path to a .ico file used for the compiled executable and installer graphics. Must be a valid Windows icon file (multi-resolution recommended: 16, 32, 48, 256 px).
  • Support Email — Contact email injected into the compiled app before each build via build_meta.json. The app reads this at startup and stores it in its settings database, so the support page shows the correct address without modifying any source files.
Icon tip: Convert a PNG to ICO using a free tool like IcoFX or the online converter at convertio.co. A multi-resolution ICO looks crisp at every Windows DPI setting.

New App Button & App Name Lookup

The New App button (in the Project Profile bar, before the Load button) clears the App Name, Exe Name, and Description fields while leaving every other setting intact. Use it to start configuring a brand-new application without losing your build environment settings (Python path, output dir, protection mode, etc.).

After clicking New App and typing in the App Name field, the dashboard automatically searches your connected GitHub repositories for a matching repo name:

  • Names are compared case-insensitively with hyphens, underscores, and spaces treated as equivalent (e.g., job-tracker matches job_tracker).
  • On a match, App URL, GitHub Remote URL, and Support URL are all filled in automatically from the matched repo's metadata.
  • If no match is found after 600 ms of inactivity, a yellow warning appears below the field: "No GitHub repo found matching this name". You can still continue — the warning is informational only.
The repo lookup requires a valid GitHub auth session (see GitHub Auth). If the badge shows Not signed in, sign in first — the lookup will then work automatically.

Code Protection

The Code Protection card controls how your Python source code is handled before it is bundled into an executable. Three options are available. Choose based on your distribution requirements and how much compile time you are willing to accept.

Protection selector
Plain
PyArmor
Nuitka

Plain — No Protection

Your .py source files are bundled directly by PyInstaller without any transformation. The resulting executable includes compiled .pyc bytecode, which is trivially reversible with tools like uncompyle6.

Use when: the project is open-source, you are building for personal use, or build speed is the top priority. Plain builds are the fastest — typically under two minutes.

PyArmor — Source Obfuscation

PyArmor encrypts and obfuscates your Python source files before PyInstaller bundles them. Each .py module is replaced with an encrypted bytecode capsule; the decryption key is bound to the PyArmor runtime that ships inside the bundle. Reverse engineering is significantly harder compared to plain bytecode.

Use when: you are distributing a commercial or proprietary tool and want a meaningful barrier against casual inspection. Build times add roughly 30–60 seconds compared to Plain.

Install PyArmor once with Install Dep (see Build Actions). You do not need to reinstall it between builds.

Nuitka — Native Machine Code

Nuitka is a Python-to-C transpiler. It converts your Python source into native C code and compiles it to a .exe directly — no Python interpreter is embedded. The resulting binary is the strongest protection available and often 20–40% faster at runtime than a PyInstaller bundle.

Use when: you need maximum performance or the highest level of IP protection. Nuitka builds take several minutes (5–20+) depending on project size.

Visual Studio Community required. Nuitka calls the MSVC C/C++ compiler (cl.exe) to compile the generated C code. Install Visual Studio Community with the Desktop development with C++ workload before clicking Full Build with Nuitka selected. The build will fail with a "cannot find cl.exe" error if MSVC is missing.
First-time Nuitka build: Nuitka downloads and caches C header stubs on its first run. Expect an extra 2–5 minutes on the initial compilation. Subsequent builds reuse the cache and are significantly faster.

License File

The License File card lets you attach a software license to your build. The license is written to LICENSE.txt in the project root and compiled into the Inno Setup installer, where it is shown on the "License Agreement" page during installation.

License card
Template:

How it works

  • Select a template from the dropdown and click Apply. The dashboard fetches the canonical license text from its internal database and writes it to LICENSE.txt in the working tree root.
  • The {year} and {publisher} placeholders in the template are automatically substituted from the App Info fields (Year defaults to the current calendar year; Publisher is taken from the Publisher field).
  • The text area is fully editable after applying — make any additional customisations before the next build.
  • The Inno Setup script picks up LICENSE.txt automatically — no further action is needed. End users see the license text on the install wizard's "License Agreement" screen.

Template categories

Protective

  • AS-IS (No Warranty) — Minimal disclaimer releasing the author from all liability. Suitable for personal or internal tools.
  • Proprietary / All Rights Reserved — Restricts redistribution and reverse engineering. Suitable for paid or closed-source products.

Open Source

  • MIT — Permissive. Allows use, copy, modify, and redistribute with attribution.
  • Unlicense — Public domain dedication. No conditions whatsoever.
  • Apache 2.0 — Permissive with patent protection clause.
  • BSD 2-Clause — Permissive. Requires attribution in binary and source forms.
  • ISC — Functionally equivalent to MIT, shorter text.

Commercial

  • Commercial (Custom) — Clears the text area with a starter commercial license skeleton. Fill in your specific terms.

Cloud refresh button

The cloud-download button next to Apply fetches the latest canonical SPDX license texts for all open-source templates directly from the SPDX database and updates the local cache. Use this if a template text looks out of date or if a new license version has been published. Proprietary and Commercial templates are not affected — their texts live only in the local database.

If you are distributing a commercial product, pair the Proprietary template with the Nuitka or PyArmor protection option for a complete IP-protection strategy.

README Generator

The README Generator card produces a README.md file for the target project using the metadata already entered in App Info. It checks whether a README.md exists in the working tree and generates or overwrites it with a structured Markdown document.

Generated content

The generated README includes:

  • Project title and description (from App Name and Description)
  • Version badge and platform badge (Windows)
  • Installation section linking to the built installer (uses App URL)
  • License reference (from the selected license template)
  • Publisher and support contact (from Publisher and Support Email)

Check vs Generate

  • Check () — Inspects the working tree for an existing README.md and reports its size and last-modified date. Does not modify any files.
  • Generate — Writes a new README.md to the project root. If a file already exists, a confirmation prompt is shown before overwriting.
Run Check before Generate to confirm you won't accidentally overwrite a hand-crafted README. After generating, open the file in your text editor to add screenshots, usage examples, and any project-specific detail.

Git — Target Project

The Git card tells the Build Dashboard which repository to build. It is separate from the Build Dashboard's own repository. All build pipeline steps operate on the project specified here.

Fields

  • Working Tree Path — Absolute path to the root of the target Git repository on this machine. The build pipeline runs all commands (git checkout, pip install, Nuitka, etc.) relative to this directory. Use the Browse button to navigate to the folder in a dialog.
  • GitHub Remote URL — The https:// remote URL for the repository. Used by the PR card and the Repo Picker. A green pulsing dot appears when the URL is reachable and your GitHub auth token has access; a red dot indicates an auth or network problem. The icon opens the Repo Picker modal.

GitHub Auth Badge

Below the Remote URL field you will see a badge showing your GitHub authentication status. A Signed in as TechnoSage badge means the GitHub CLI (gh) is authenticated and the Build Dashboard can make API calls on your behalf. A Not signed in badge means you need to authenticate — see GitHub Auth.

The Working Tree Path must point to the development branch in normal usage. The build pipeline automatically checks out main for the production build and switches back to development when finished.

Git Operations

The Git card exposes quick-action buttons for common repository operations on the target project — the repo at Working Tree Path. These run git commands directly against that repo (not against the Build Dashboard's own repository).

Git Status & Refresh

The refresh button () in the Git card header fetches the current branch name, working-tree status (clean / modified / ahead), and the last commit message. Results appear in the status area below the URL field.

Commit & Push

The Commit & Push button stages all changed files, creates a commit with the message you enter, and pushes to the origin remote on the current branch. The operation runs against the Working Tree Path and uses whatever branch is currently checked out. A success or error message is appended to the Build Log.

Commit & Push stages all modified files (equivalent to git add -A). Review your working tree before using this button if you have files you do not want to commit. Use your normal git client for selective staging.

Push Branch

Pushes the current branch to origin without committing. Use this to update the remote after a local commit made via your usual git tooling. If the remote branch does not exist yet, it is created (git push --set-upstream origin <branch>).

Git Setup

The Git Setup action initialises the repository if it is not already a git repo, sets the remote URL to the value in GitHub Remote URL, and configures the local user name and email from your system git config. Run this once when pointing the dashboard at a brand-new project folder that has not yet been git-initialised.

Merge

Merges a specified source branch into the currently checked-out branch. The branch name is entered in the input field that appears when you click the merge action. A fast-forward merge is attempted first; if that fails (diverged history), a standard merge commit is created. Merge conflicts are reported in the Build Log and must be resolved manually.

Pipeline Sounds

The Pipeline Sounds card (above the CI/CD Pipeline) lets you assign audio cues to build and test results. A Success sound plays when either pipeline completes without errors; a Fail sound plays when a stage errors out. Sounds fire automatically on real builds and tests — not on simulation button presses.

Choosing a sound

Each sound field (Success / Fail) has three ways to select a file:

  • Type a path — paste any absolute path directly into the text field.
  • System sounds dropdown — click the split-button caret () to scan your computer for audio files in %WINDIR%\Media and Windows Themes directories. The list is grouped by folder and cached for the session.
  • Browse — opens a Windows file picker filtered to common audio formats (.wav .mp3 .ogg .m4a .flac .wma). Defaults to C:\Windows\Media.

Preview

Click the play button ( or ) next to each field to preview the configured sound immediately. A toast notification appears if the file cannot be played.

Supported formats: WAV files play via the Windows system sound API (non-blocking, near-instant). MP3, OGG, FLAC, and other formats play via Windows Media Player through PowerShell — allow up to 1–2 seconds for playback to start.

CI/CD Pipeline

The visual pipeline strip sits directly above the build log. It shows the five sequential stages of every full build and updates in real time as each stage completes.

Pipeline — mid-build state
Deps
Tools
Protect
Bundle
Installer

Stage descriptions

  • Deps — Installs Python dependencies from requirements.txt using pip install -r. Also confirms that all required external tools (Inno Setup, PyInstaller) are on the PATH.
  • Tools — Verifies Inno Setup (iscc.exe) and the selected protection tool are accessible. Checks the Git working tree is clean and checks out the main branch for the build.
  • Protect — Runs the selected code protection step. For Plain this stage is a no-op. For PyArmor it obfuscates source into obf_src/. For Nuitka it compiles Python to native C and produces dist/.
  • Bundle — Runs PyInstaller (or Nuitka's built-in bundler) to produce a self-contained .exe or folder in dist/. All assets, templates, and static files are included via the spec file.
  • Installer — Runs Inno Setup Compiler (iscc.exe) against the generated .iss script. The output installer is written to the Output Dir configured in App Info.

Visual feedback

  • 3D tubes with flowing particles — Each stage tube fills with a neon-green gradient as it progresses. Glowing particles flow through the arrows between stages.
  • 3D flags — A Start flag lights up when a build begins; a Finish flag (checkered) lights up when all stages complete.
  • Red pulse — If any stage fails, the failed tube pulses red and the pipeline halts. Error details appear in the Build Log below.
  • Race car + fireworks — The celebration animation plays only when both the CI/CD Pipeline and the Test Pipeline complete successfully without errors.
  • BUILD — BUILD simulation buttons — Run and Error buttons in the pipeline card header simulate a success or failure animation without running an actual build.
  • TEST — TEST simulation buttons — TP Run and TP Error buttons next to the build simulations drive the Test Pipeline animation independently, for UI testing.

Auto-open output folder

After a successful Bundle Only or Full Build, Windows Explorer automatically opens to the output directory (dist/ for bundle, configured Output Dir for full build) so you can immediately inspect or distribute the result.

Clean Outputs resets both the CI/CD Pipeline and Test Pipeline to their initial state, clearing all stage fills and flags before you start a fresh build.

Virus / Malware Scan

Before any code is compiled, Build Dashboard automatically runs a Windows Defender custom scan against the entire project source directory. This ensures that no infected or tampered file enters the compiled binary or installer.

Pipeline — Scan stage (first stage)
Scan
→
Deps
→
Tools
→
Protect
→
Bundle
→
Installer

How it works

  • The scan runs against the Working Tree Path (the full source directory) using MpCmdRun.exe -Scan -ScanType 3 -File <source> — a targeted custom scan rather than a full system scan, so it completes in seconds to minutes depending on project size.
  • Clean result (exit 0) — Build continues normally. The Build Log shows: [OK] Scan complete — no threats detected.
  • Threat found (exit 2) — Build halts immediately. An [ERROR] message is written to the log and the pipeline stage turns red. You must quarantine or remove the threat and rebuild.
  • Defender not installed / scan error — A [WARN] is written and the build continues. This is non-blocking so the dashboard works on machines without Defender, but a manual scan is strongly recommended before distribution.

Windows Defender location

Build Dashboard searches for MpCmdRun.exe in three locations (in order):

  1. C:\Program Files\Windows Defender\MpCmdRun.exe — static path on all Windows versions
  2. C:\ProgramData\Microsoft\Windows Defender\Platform\<version>\MpCmdRun.exe — versioned path on Windows 10/11 after a Defender engine update
  3. System PATH — as a fallback for non-standard installations
Keep Windows Defender definitions up to date for the most effective pre-build scan. Run Windows Security → Virus & threat protection → Check for updates before a release build.
The scan uses -DisableRemediation — it detects threats but does not quarantine or delete them automatically. If a threat is found, use Windows Security or MpCmdRun.exe -Remove to remediate before rebuilding.

Build Actions

Four action buttons control the build pipeline. They appear in a row above the pipeline strip.

Action buttons

Clean Outputs

Deletes all build artefact directories. Specifically:

  • build_output/ and obf_src/ inside the project root (PyInstaller work tree, obfuscation intermediates).
  • The entire configured Output Dir tree (e.g. D:\Compile Playground\job_tracker\), which contains Bundle Only\, Log\, and any installer .exe files.

Use this before a full rebuild to ensure no stale artefacts from previous builds are included. A confirmation dialog is shown because this action cannot be undone.

Bundle Only

Runs only the Protect and Bundle stages — skipping the Deps, Tools verification, and Installer stages. Use this to verify bundling works or to produce a standalone exe without creating an installer. The output lands in <OutputDir>\Bundle Only\<ExeName>\ (e.g. D:\Compile Playground\job_tracker\Bundle Only\JobTracker\). On success, Windows Explorer opens to the Output Dir automatically, and the Test Pipeline runs to verify the bundle starts and responds on the configured port.

Full Build

Runs all five pipeline stages in sequence: Deps → Tools → Protect → Bundle → Installer. This is the normal path for producing a distributable installer. The Git workflow is:

  1. Stash or confirm the working tree is clean on development.
  2. Checkout main branch.
  3. Run all build stages.
  4. Return to development branch.

On success, Windows Explorer opens to the configured Output Dir automatically showing the finished <ExeName>_Setup_<Version>.exe, and the Test Pipeline launches to verify the installer silently installs and the app responds on its port.

Do not close the browser tab or shut down the build server while a Full Build is running. Doing so may leave the repository checked out on main and require a manual git checkout development to recover.

Install Dep

Installs the currently selected protection tool (PyArmor or Nuitka) into the project's virtual environment via pip install. Run this once after selecting a new protection strategy or after setting up the project on a new machine. The button has no effect when Plain is selected.

Auto-Kill Running App Before Build

When Bundle Only or Full Build is clicked, the dashboard automatically kills any running instance of the target app (identified by Exe Name) before the compile starts. This prevents file-locking errors where the bundler cannot overwrite .exe or .pyd files that are held open by the running process. The kill is logged at the top of the Build Log so you can confirm it happened. You can also trigger a manual kill at any time via the Kill App button if visible in the toolbar.

Inject Metadata

Before each build, the dashboard writes a build_meta.json file into the project source root containing:

  • Support Email — the address from the App Info card
  • Version — current version string
  • App Name and Publisher
  • Build timestamp

The target app reads build_meta.json at startup and persists these values to its own database. This means you can update the support email or version in the dashboard and rebuild — no source-file edits needed. The Inject Meta action can also be triggered standalone (without a full build) to push updated metadata immediately.

Version Management

The Version field in the App Info card has dedicated increment controls so you never need to hand-edit the version string:

  • + Major — Increments the first number and resets Minor and Patch to zero (e.g., 1.4.2 → 2.0.0). Use for breaking changes.
  • + Minor — Increments the second number and resets Patch to zero (e.g., 1.4.2 → 1.5.0). Use for new features.
  • + Patch — Increments the third number (e.g., 1.4.2 → 1.4.3). Use for bug fixes.

Each increment is saved to build_settings.json immediately. The updated version string is embedded in the compiled executable and installer on the next build.

Bump the version before running a Full Build, not after. The version embedded in the installer is read from build_settings.json at build time.

Build Log

The Build Log is a scrolling terminal-style panel below the pipeline strip. It streams output from every subprocess the build pipeline runs, in real time via a server-sent events (SSE) connection.

Build log — colour codes
[OK] Installed 12 packages from requirements.txt
[INFO] Running PyArmor obfuscation on 8 modules...
[WARN] Large file detected: models.py (2.3 MB) — compile time may be high
[ERROR] iscc.exe not found on PATH — install Inno Setup first
── ── ── ── Stage: Bundle ── ── ── ──

Colour codes

  • Green — Success or confirmation message. The step completed without issues.
  • Grey / white — Informational output. Normal subprocess stdout.
  • Yellow — Warning. The build continues but something may need attention.
  • Red — Error. The pipeline halts at the current stage. Read this line to diagnose the problem.
  • Dark grey separator — Stage boundary marker. Makes it easy to find where each stage starts and ends.

Controls

  • Open Output Folder — The folder icon in the log card header opens the Output Dir in Windows Explorer so you can immediately inspect or distribute the built installer.
  • Auto-scroll — The log auto-scrolls to the bottom while a build is running. Manually scrolling up pauses auto-scroll; it resumes when you scroll back to the bottom.
  • Clear — Clears the log panel contents. Does not affect any build in progress.
If a build fails, the error line in red is your starting point. Copy the message and search for it alongside the tool name (e.g., "Nuitka cannot find cl.exe") for targeted troubleshooting.

Test Pipeline

The Test Pipeline (cyan/blue colour scheme) sits below the Build Log and runs automatically 1.5 seconds after a successful Bundle Only or Full Build. It launches the compiled output in the background, polls the app port, and verifies the app responds correctly — giving you immediate confidence before distributing.

Stages

  • Launch — Starts the compiled exe (bundle test) or runs the installer silently (full build test) in a background process.
  • Connect — Polls http://127.0.0.1:{port} every second until the app responds or a timeout is reached.
  • Verify — Confirms the HTTP response is valid (status 200). Checks that the app is not just listening but actually serving content.
  • Report — Writes the test result to the Test Output Log. On success, triggers the race car + fireworks celebration (if the build pipeline also passed).

Timeouts

  • Bundle Only test — up to 20 seconds for the app to respond.
  • Full Build test — up to 25 seconds (extra time for installer execution).

Visual feedback

The Test Pipeline uses the same 3D tube + flag animation as the CI/CD Pipeline, overridden to cyan and blue tones. The card glows cyan while a test is running. A TP Run and TP Error button in the CI/CD Pipeline card header let you simulate test success and failure for UI testing without actually running a build.

Celebration gate

The race car + fireworks animation plays only when both the CI/CD Pipeline and the Test Pipeline finish successfully. If the build passed but the test failed (app did not start), no celebration is shown, signalling the output needs investigation.

A configured Success sound (Pipeline Sounds card) plays when the Test Pipeline completes cleanly; the Fail sound plays when a test stage errors out.

Test Output Log

The Test Output Log card sits below the Test Pipeline and displays the streaming output from test runs — app startup messages, port polling results, and any error tracebacks from the tested executable.

The log is automatically populated when the Test Pipeline runs after a real build. It shows the same colour-coded output as the Build Log: green for success confirmations, grey for informational output, and red for errors.

Persistent log files

Every test run writes a timestamped .log file to <OutputDir>\Log\ (e.g. D:\Compile Playground\job_tracker\Log\JobTracker_Output_Pipeline_2026-03-12_14-30-00.log). The file includes a structured header (app name, mode, result, timestamp) and a full transcript of every test step. Open the Output Dir in Explorer and browse the Log\ subfolder to review historical test results.

Most common failure cause: if the compiled exe "does nothing" (no port response), the app is likely crashing silently on startup. Check the log shown here — the last traceback line is usually enough to pinpoint the problem. A frequent culprit is a missing WorkingDir entry in the Inno Setup [Run] section, which causes relative-path lookups (templates, database) to fail.
If the log is empty after a test failure, the process likely crashed before Flask initialised. Run the exe manually from a terminal to capture raw stderr output.

HTTPS & Secure Connections

By default Flask serves over plain http://, which causes modern browsers (Chrome, Edge) to display a "Not Secure" warning in the address bar. Build Dashboard and Job Tracker both support HTTPS via locally-trusted certificates generated by mkcert — a zero-configuration tool that creates certificates signed by a private CA that it installs directly into the Windows, Chrome, and Firefox trust stores.

HTTPS here is for local loopback traffic only (127.0.0.1 / localhost). No data leaves your machine — this purely removes the browser warning and hardens the connection between your browser and the local Flask server.

Once certificates exist in the certs/ folder next to the app, both run.py (Job Tracker) and start_builder.py (Build Dashboard) detect them automatically and start Flask with ssl_context set. The browser is then opened with https:// instead of http://.

Step 1 — Install mkcert

mkcert is a small open-source tool (BSD 3-Clause licence). Install it once per developer machine:

PowerShell — install via Winget (recommended)
winget install FiloSottile.mkcert

Or download the latest mkcert-v*-windows-amd64.exe from the mkcert releases page, rename it to mkcert.exe, and place it somewhere on your PATH (e.g. C:\Windows\System32 or a folder already in PATH).

After installing, run the one-time CA setup in an Administrator PowerShell:

PowerShell (run as Administrator)
mkcert -install

This creates a private CA key in %LOCALAPPDATA%\mkcert\ and registers it as trusted with Windows, Chrome, and Firefox. You only need to do this once.

Step 2 — Generate Certificates

Create a certs/ folder inside the project root, then run mkcert to generate certificates for localhost:

PowerShell — from the project root (e.g. job_tracker/)
mkdir certs
cd certs
mkcert 127.0.0.1 localhost

This produces two files:

  • certs\localhost.pem — the certificate (public)
  • certs\localhost-key.pem — the private key (keep secret, never commit)
Both files are listed in .gitignore so they are never committed to version control. The private key must stay on your machine only.

Once the files are in place, restart the app (python run.py or python start_builder.py) and open https://127.0.0.1:5000 (or 5001). Chrome should show a padlock icon with no warnings.

Certificate Renewal

mkcert certificates are valid for 2 years + 3 months by default. When they expire the browser will show a certificate error. Simply regenerate them:

PowerShell — from project root
cd certs
del localhost.pem localhost-key.pem
mkcert 127.0.0.1 localhost

Restart the app after regenerating — no other steps needed.

The expiry date is embedded in the cert. Run mkcert -CAROOT to find where your CA root lives, or inspect the cert with openssl x509 -in certs\localhost.pem -noout -dates.

Bundling HTTPS in Compiled Installers

When you build an installer via Build Dashboard, mkcert can be bundled and run automatically on the end-user's machine as part of the install process. This is handled in build/build.py:

  • If mkcert.exe is found on your PATH at build time, it is automatically included in the Inno Setup [Files] section.
  • A Pascal [Code] block runs mkcert -install (to register the CA) and then mkcert 127.0.0.1 localhost (to generate certs into {app}\certs\) after the main files are copied.
  • Since Inno Setup installers run as Administrator by default, the CA registration succeeds without any extra UAC prompts.
Licensing: mkcert is BSD 3-Clause licensed and may be freely redistributed with compiled applications. The generated certificates and CA root are machine-specific and are not covered by any software licence.
The mkcert CA is trusted by the user's machine and browser. It can only issue certificates for domains/IPs you specify — it does not grant any access to external websites or the public internet.

Local Backup

The Local Backup card provides a built-in robocopy-based backup solution that mirrors your project source to a secondary location (e.g. a separate drive). This is independent of Git and provides a fast, versioning-free safety net.

Local Backup card
Source Folder
C:\...\job_tracker
Destination Folder
D:\backups\job_tracker
Auto Backup

How it works

Clicking Backup Now runs:

Command executed
robocopy "C:\...\job_tracker" "D:\backups\job_tracker" /MIR /R:2 /W:3 /NP /LOG+:backup.log
  • /MIR — Mirror mode: files deleted from the source are also deleted from the destination, keeping the two perfectly in sync.
  • /R:2 /W:3 — Retry failed files twice with a 3-second wait (handles file-lock conflicts).
  • /NP — No progress percentage in output (keeps the build log clean).
  • /LOG+ — Appends robocopy output to backup.log in the project root.

Auto Backup schedule

  • Manual only — Backup only runs when you click Backup Now.
  • On app open — A single backup runs each time the Build Dashboard starts. No daemon is required.
  • On app open + every 5 / 10 / 30 min — Runs once on startup, then continues on the selected timed interval via the Build Dash Daemon.
  • Every 5 / 10 / 30 minutes — Timed interval only (no startup backup). The Build Dash Daemon (see below) runs the backup on the selected interval, even when the browser tab is closed.

Status badge

A status badge next to the Backup Now button reflects the current backup state: Idle (no backup has run yet), Running (robocopy in progress), Success (last backup completed without errors), or Failed (robocopy returned a non-zero exit code). The badge also shows the time of the last run.

/MIR will delete files in the destination that no longer exist in the source. Ensure the destination folder is dedicated to this backup and does not contain other files you want to keep.
Set the destination to a folder on a separate physical drive (e.g. D:\backups\) for protection against OS-drive failure. Network shares (UNC paths) are also supported by robocopy if the share is always accessible.

Build Dash Daemon

The Build Dash Daemon is a background Windows process that runs independently of the browser session. Its primary job is to execute scheduled backups on the configured interval, but it can also perform other background maintenance tasks. When the daemon is running, backups happen even if you have closed the Build Dashboard tab.

Daemon card
Tray Icon (.ico)
build/assets/daemon.ico
App Name (PyInstaller)
build_dash_daemon
Daemon running PID 18432

Configuration fields

  • Tray Icon (.ico) — Path to the .ico file used as the daemon's system tray icon. When the daemon is compiled with PyInstaller (Build .exe), this icon is also embedded in the daemon executable itself.
  • App Name (PyInstaller) — The --name argument passed to PyInstaller when building the daemon executable. The resulting file is <App Name>.exe in build_output/.

Start / Stop

Start launches scheduler_daemon.py (or the compiled daemon .exe if it exists) as a detached background process. The dashboard polls for the PID file (scheduler_daemon.pid) written by the daemon to determine whether it is running. The status indicator shows a green pulsing dot when the PID file is present and the process is alive.

Stop sends a termination signal to the PID recorded in the PID file and deletes the file. If the PID file is stale (the process already exited), Stop cleans up the file and resets the status indicator.

Build .exe

Click Build .exe to compile scheduler_daemon.py into a standalone .exe using PyInstaller (Plain bundling — the daemon has no code protection option). The output lands in build_output/ alongside the main application installer. Once compiled, clicking Start will prefer the compiled .exe over the raw Python script.

The daemon is only required for automatic scheduled backups. If you prefer to run backups manually by clicking Backup Now, you do not need to start or build the daemon at all.
Add the compiled daemon .exe to Windows Task Scheduler or the Startup folder if you want backups to resume automatically after a system reboot, without needing to open the Build Dashboard first.
